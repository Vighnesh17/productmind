  ProductMind AI — Greenfield Architecture Plan

  Managed Services Strategy

  The core principle: your local environment and production environment use the same services, just different tiers. No "we use X locally
   and Y in production" mismatches.

  COMPONENT          LOCAL                        PRODUCTION
  ────────────────────────────────────────────────────────────────────
  Database           Supabase CLI (local Docker)  Supabase Cloud (managed PG)
  Background Jobs    Inngest Dev Server           Inngest Cloud
  Redis              Upstash (free tier)          Upstash (paid tier)
  Vector DB          Pinecone (free tier)         Pinecone (paid tier)
  Auth               Auth0 (dev tenant)           Auth0 (prod tenant)
  Secrets            .env + Doppler CLI           Doppler (prod env)
  Frontend           Next.js dev server           Vercel
  Backend API        uvicorn (hot reload)         Cloud Run
  Monitoring         Terminal logs                Better Stack
  Storage            Supabase Storage (local)     Supabase Storage (cloud)

  Key swap from original plan: Celery → Inngest.
  Inngest is a managed background job service with a local dev server (inngest dev spins up a dashboard at localhost:8288). It
  eliminates:
  - The Celery + Redis broker complexity
  - The Cloud Run cold-start / min-instances problem
  - Manual retry/backoff implementation
  - The need for a separate worker service

  The API service calls Inngest to enqueue work. Inngest calls back to the API via HTTP. Local and cloud work identically.

  Agent Execution Flow

  PM sends message
        │
        ▼
  POST /api/v1/agent/{id}/message
        │
        ├─── Auth0 JWT validation
        ├─── Tenant scoping (tenant_id from JWT, never from body)
        ├─── Budget check (Upstash Redis counter)
        │    └─ Over budget → 429 immediately
        │
        ├─── Persist message to Supabase (conversations.messages)
        ├─── Enqueue Inngest job: "agent/execute"
        │
        └─── Return: { task_id, status: "queued" }
                     + open SSE channel for streaming updates

  Inngest receives job
        │
        ▼
  agent/execute function
        │
        ├─── Load conversation history (Supabase)
        ├─── Recall relevant context (Pinecone semantic search)
        │    └─ Timeout → continue with empty context, log metric
        │
        ├─── Fetch OAuth tokens (Doppler / Secret Manager)
        │
        ├─── Claude API call (streaming, tool_use enabled)
        │    │
        │    └─── Tool use loop:
        │         ├─── Validate tool name (registry lookup)
        │         ├─── Validate params (Pydantic)
        │         ├─── Execute tool (Jira/Slack API)
        │         └─── Return result to Claude
        │
        ├─── Stream chunks to SSE via Upstash Redis pub/sub
        │
        ├─── Persist: final message to Supabase (source of truth)
        ├─── Upsert: embeddings to Pinecone (async, non-blocking)
        └─── Decrement: token budget in Upstash Redis

  Agent Executor State Machine

                      ┌──────────┐
                      │  QUEUED  │ ◄─── Inngest job received
                      └────┬─────┘
                           │
                           ▼
                      ┌──────────┐
                      │ CONTEXT  │ ◄─── Pinecone recall
                      │ LOADING  │      Timeout → DEGRADED (continues)
                      └────┬─────┘
                           │
                           ▼
                ┌──────────────────────┐
                │     LLM CALLING      │ ◄─── Claude tool_use
                │  (streaming tokens   │      Timeout → RETRY (2x)
                │   to SSE via Redis)  │      Rate limit → RETRY (backoff)
                └──────┬───────────────┘      Exhausted → FAILED
                       │
                ┌──────▼───────────────┐
                │   TOOL EXECUTING     │ ◄─── Jira/Slack API calls
                │  (loop until no      │      Unknown tool → log, skip
                │   more tool_use)     │      API error → inform Claude
                └──────┬───────────────┘
                       │ final response
                       ▼
                      ┌──────────┐
                      │PERSISTING│ ◄─── Supabase write (must succeed)
                      └────┬─────┘      Pinecone write (best-effort)
                           │
                           ▼
                      ┌──────────┐
                      │COMPLETED │ ──── SSE "done" event sent
                      └──────────┘

    FAILED state: SSE "error" event sent, message saved with error status
    Impossible: COMPLETED → any state (terminal)
    Impossible: QUEUED → TOOL_EXECUTING (must go through LLM first)

  Architecture Issues

  Issue 1 — OK: Inngest replaces Celery. No worker service to deploy. No Redis broker to manage. Inngest handles retries, backoff,
  idempotency, and observability. The API service is the only backend service. Cloud Run min-instances problem from the CEO review is
  eliminated entirely.

  Issue 2 — OK: Supabase local CLI for dev. supabase start spins up Postgres, Auth, and Storage locally via Docker with the exact same
  RLS policies as production. Zero configuration drift between environments.

  Issue 3 — WARNING: SSE streaming via Redis pub/sub. Inngest calls back to the API via HTTP (webhook pattern). The API needs to forward
  streaming chunks to the browser SSE connection. Since Cloud Run instances are stateless (different instance may receive the Inngest
  callback vs. the one holding the SSE connection), chunks must route through Upstash Redis pub/sub. Each SSE connection subscribes to a
  task:{task_id} channel.

  Browser SSE ←── FastAPI instance A ←── Redis sub("task:abc")
                                                ▲
  Inngest callback ──► FastAPI instance B ──── Redis pub("task:abc", chunk)

  This works correctly but must be explicit in the implementation.

  Issue 4 — WARNING: Doppler for secrets. Using Doppler instead of GCP Secret Manager gives identical developer experience locally and in
   production (doppler run -- uvicorn ...). No .env files committed. Every engineer pulls from Doppler automatically. Doppler syncs to
  GCP Secret Manager in production if needed, or Cloud Run can pull from Doppler directly via the Doppler CLI in the Docker image.

  Required Code Patterns

  1. Tool Registry pattern (tools/registry.py):
  # All tools registered at startup. Agent loop validates tool names here.
  # Unknown tool name → ToolNotFoundError immediately, never reaches Claude.
  REGISTRY: dict[str, BaseTool] = {}

  def register(tool: BaseTool) -> None:
      REGISTRY[tool.name] = tool

  def get(name: str) -> BaseTool:
      if name not in REGISTRY:
          raise ToolNotFoundError(f"Unknown tool: {name}")
      return REGISTRY[name]

  2. All prompts in markdown files, never inline:
  # agents/pm/agent.py
  SYSTEM_PROMPT = Path("agents/pm/prompts/system.md").read_text()
  # Never: system_prompt = "You are a PM agent that..."

  3. tenant_id always from JWT, set via db context:
  # db/rls.py
  async def set_tenant_context(session: AsyncSession, tenant_id: str) -> None:
      await session.execute(
          text("SELECT set_config('app.current_tenant_id', :tid, true)"),
          {"tid": tenant_id}
      )
  # This is called in every request before any DB query.
  # tenant_id is NEVER accepted from request body or path params.

  4. Structured logging — every log has context:
  import structlog
  log = structlog.get_logger()

  # Always bind context at the top of each function:
  log = log.bind(tenant_id=tenant_id, agent_id=agent_id, task_id=task_id)
  log.info("agent.tool.executed", tool=tool_name, latency_ms=elapsed, success=True)

  5. Config from Pydantic Settings (Doppler → env vars → config):
  # config.py
  class Settings(BaseSettings):
      anthropic_api_key: str
      supabase_url: str
      supabase_anon_key: str
      pinecone_api_key: str
      upstash_redis_url: str
      auth0_domain: str
      auth0_audience: str
      inngest_signing_key: str
      doppler_token: str | None = None  # Optional, used by CLI

      model_config = SettingsConfigDict(env_file=".env")

  settings = Settings()

  N+1 risk: When loading conversation history, eager-load messages with the conversation. Never for msg in conversation.messages without
  preloading.

  Caching strategy (Upstash Redis):
  jira:sprint:{tenant_id}:{sprint_id}      TTL: 5min
  jira:ticket:{tenant_id}:{ticket_id}      TTL: 5min
  slack:channel:{tenant_id}:{channel_id}   TTL: 15min
  budget:{tenant_id}:{date}                TTL: until midnight (rolling)

  Memory concern: Slack channel history for 30 days can be large. Fetch in pages, summarise progressively, never load all messages into
  memory at once.

  CODEPATH                   | FAILURE                  | TEST? | HANDLED? | USER SEES?
  ──────────────────────────────────────────────────────────────────────────────────────
  JWT validation             | Invalid/expired token     | YES   | YES      | 401

  CRITICAL GAPs requiring fixes before Phase 1 ships:

  1. Claude empty response: Add explicit check after streaming. If response has no content blocks → retry once with "please provide a
  response" → if still empty → AgentRefusalError → user sees "AI gave no response, please retry."
  2. Supabase write failure: Wrap in try/except sqlalchemy.exc.OperationalError. Retry once (100ms delay). If still failing → Inngest
  marks job as failed → requeue. User sees loading state until resolved.
  3. SSE Redis pub/sub failure: If Redis pub/sub channel can't be established → fall back to polling (GET /api/v1/task/{id}/status).
  Frontend should implement polling fallback automatically after 5s with no SSE events.
  4. Budget check when Redis is unavailable: Fail open (allow the request) and log budget.check.unavailable metric. Revenue > budget
  enforcement during an outage.
  5. Doppler/OAuth token unavailable: Agent task fails immediately with IntegrationAuthError. User sees "Your Jira connection is
  unavailable — please reconnect." This must not silently produce an empty sprint report.

  Local → Production Phasing

  Local Dev Commands (Day 1 setup)

  # 1. Install tools
  brew install supabase/tap/supabase doppler inngest

  # 2. Start local Supabase
  supabase start
  # → Postgres at localhost:5432
  # → Studio at localhost:54323

  # 3. Run migrations
  supabase db push

  # 4. Start Inngest dev server
  inngest dev
  # → Dashboard at localhost:8288

  # 5. Pull secrets from Doppler
  doppler setup  # links to your Doppler project

  # 6. Start backend
  doppler run -- uvicorn backend.api.main:app --reload --port 8000

  # 7. Start frontend
  cd frontend && npm run dev
  # → localhost:3000

  One terminal per service. No docker-compose required for day-to-day dev. supabase start handles the Postgres container internally.

  Production Deploy (Cloud Run + Vercel)

  DEPLOY SEQUENCE:
    Step 1: Push Supabase migrations
            supabase db push --linked
            → verify RLS policies active

    Step 2: Build and push Docker image (API only — no worker)
            docker build -f Dockerfile -t gcr.io/{project}/api:{sha} .
            docker push gcr.io/{project}/api:{sha}

    Step 3: Deploy Cloud Run
            gcloud run deploy productmind-api \
              --image gcr.io/{project}/api:{sha} \
              --set-secrets=... (from Doppler or GCP Secret Manager)
            → smoke: GET /health → 200

    Step 4: Register Inngest functions (automatic on first request)
            curl https://api.productmind.ai/api/inngest
            → Inngest discovers and registers all functions

    Step 5: Deploy frontend
            vercel --prod
            → smoke: load page, verify auth flow

    Step 6: E2E smoke test
            → Login → create agent → connect Jira test account → sprint status report

  Rollback:
  - API: Cloud Run revisions → route 100% traffic to previous revision (<2 min)
  - Frontend: Vercel → Deployments → instant rollback (<1 min)
  - Database: All Phase 1 migrations are additive (no drops, no renames) → no migration rollback needed
